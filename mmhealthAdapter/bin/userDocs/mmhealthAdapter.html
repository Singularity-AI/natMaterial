Introduzione

  1. Why would we make the effort to split up our codebase into multiple modules when everything works just fine with a single, monolithic module?
    The main reason is that a single monolithic codebase is susceptible to architectural decay.
    Java packages aren’t very good at protecting architectural boundaries: dependencies between classes within a single monolithic codebase tend to quickly degrade into a big ball of mud.
    See Building a Multi-Module Spring Boot Application with Gradle (by Hombergs).
  2. If we split up the codebase into multiple smaller modules that each has clearly defined dependencies to other modules, we take a big step towards an easily maintainable codebase.
    We can split an application up along technical layers, or along functional boundaries, or both.
  3. It could have sense to cut a Spring application context into multiple Spring modules along the same boundaries. The base for a Spring Module is a @Configuration-annotated class along the lines of Spring’s Java configuration feature.
    See Modularizing a Spring Boot Application (by Hombergs).
  4. However, to better enforce module boundaries, we could split up a monolithic codebase into multiple build modules with Gradle.
    See Building Java Projects with Gradle or Gradle Tutorial : Part 3 : Multiple Java Projects

Start-up (prepare the Parent)

  1. Set up the working directory: 
    .
├── haAdapters
	├── haPersistence	//Perhaps empty
	├── haWeb
├── haApplication   
├── haCommon
├── haCnfiguration
├── haDomain
├── uerDocs
  2. Define the system modules in the parent build and the top-level gradle build file 
    settings.gradle
build.gradle 	//initial version that includes an explanation task for each subproject
  3. Prepare Gradle Wrapper in the parent build directory: 
    gradle wrapper --gradle-version 6.3 --distribution-type all
See the generated gradle/wrapper/gradle-wrapper.properties

  4. gradlew build	//Gradle will automatically resolve any dependencies between the modules  
				//and build them in the correct order, regardless of the order they are listed in settings.gradle.
  5. Execute the explanation task: 
    gradlew -q explain
Outpput ---------------------------------
Hello, I'm mmhealthAdapter project
Hello, I'm haAdapters project
Hello, I'm haApplication project
Hello, I'm haCommon project
Hello, I'm haConfiguration project
Hello, I'm haDomain project
  6. Add istructions specific to a subproject
    .
├── haAdapter
├── haApplication   
	└── build.gradle (just an example, at the moment)
├── haDomain
├── haLibrary
├── uerDocs
...
  7. Execute the explanation task again 
    gradlew -q explain
Outpput ---------------------------------
Hello, I'm mmhealthAdapter project
Hello, I'm haAdapters project
Hello, I'm haApplication project
- I'm the main application that uses haCommon
Hello, I'm haCommon project
Hello, I'm haConfiguration project
Hello, I'm haDomain project
  8. Execute just the project-specific build 
    gradlew -q :haApplication:explain
Outpput ---------------------------------
Hello, I'm haApplication project
- I'm the main application that uses the haCommon
    Gradle always evaluates every project of the multi-project build and creates all existing task objects.
    Then, according to the task name arguments and the current dir, Gradle filters the tasks which should be executed.
  9. Add build istructions common to all subprojects 
    build.gradle 	//initial version that defines the basic configuration that is shared across all sub-modules.
