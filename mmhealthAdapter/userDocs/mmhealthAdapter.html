Introduzione

  1. Why would we make the effort to split up our codebase into multiple modules when everything works just fine with a single, monolithic module?
    The main reason is that a single monolithic codebase is susceptible to architectural decay.
    Java packages aren’t very good at protecting architectural boundaries: dependencies between classes within a single monolithic codebase tend to quickly degrade into a big ball of mud.
    See Building a Multi-Module Spring Boot Application with Gradle (by Hombergs).
  2. If we split up the codebase into multiple smaller modules that each has clearly defined dependencies to other modules, we take a big step towards an easily maintainable codebase.
    We can split an application up along technical layers, or along functional boundaries, or both.
  3. It could have sense to cut a Spring application context into multiple Spring modules along the same boundaries. The base for a Spring Module is a @Configuration-annotated class along the lines of Spring’s Java configuration feature.
    See Modularizing a Spring Boot Application (by Hombergs).
  4. However, to better enforce module boundaries, we could split up a monolithic codebase into multiple build modules with Gradle.
    See Building Java Projects with Gradle or Gradle Tutorial : Part 3 : Multiple Java Projects

Modules

Before entering in the code, we recall that @SpringBootApplication annotation is a shortcut for a set of annotations that usually prefix the code of a Main application class in SpringBoot:
  • @Configuration to enable Java-based configuration
  • @ComponentScan to enable component scanning
  • @EnableAutoConfiguration to enable Spring Boot's auto-configuration feature
To make things more explicit, for each module, we could define a specific package name for its software. In this way, the auto-configuration feature included in the @SpringBootApplication annotation of the configuration module HealthAdapterApplication.java does not work, and we are forced to use explicit declarations (e.g. using an @Import annotation).

However, if we put the configuration class under the 'root' package it.unibo, all its sub-packages and classes will be searched during an auto-configuration phase.

The parent (root)

.
└── build.gradle

gradlew build
gradlew bootRun	// To execute
Here :
  • we declare that we use the io.spring.dependency-management SpringBoot plugin.

    Since Gradle 5+ supports BOM (Bill Of Materials) files, you don't need the dependency-management plugin anymore, but the SpringBoot plugin is still needed to provide tasks such as bootJar and bootRun.
  • we specify what all subprojects require. The only dependency we add to each subproject is to the BOM of Spring Boot.
    This BOM includes all dependencies that a Spring Boot application potentially might need in the exact version that is compatible with a given Spring Boot version (2.3.4.RELEASE in our case). Thus, we don’t need to list every single dependency on our own and potentially get the version wrong.

The Configurator module

├──haConfiguration   
  ├── src/main/java/it/unibo/mmhealthAdapterConfig	
    └──  HealthAdapterApplication.java
  └── build.gradle 	
  
gradlew :haConfiguration:build  
gradlew :haConfiguration:bootRun  // To execute
Here, in build.gradle, we specify:
  • the modules that compose the system
  • that we will use the Spring Boot plugin version 2.3.4.RELEASE
The module includes the HealthAdapterApplication.java annoted with @SpringBootApplication.

The Application module

├── haApplication   
  ├── src/main/java/it/unibo/mmhealthAdapterAppl	
    └── HealthAdapterApplLogic.java
  └── build.gradle 	
  
gradlew :haApplication:build
Here, in haApplication/build.gradle we declare :
  • the dependencies required by the business logic software
The (quite naive, at the moment) business logic is defined in HealthAdapterApplLogic.java .

The Web module

├── haAdapters
  ├── haWeb
    ├── src/main/java/it/unibo/mmhealthAdapter	
      └── HIDemoController.java
      └── MIDemoController.java
    ├── src/main/resources/templates
      └── gui.html
  └──build.gradle 	
  
  
gradlew :haAdapters:haWeb:build
Here, in haweb/build.gradle, we declare:
  • a dependency to the haApplication module, since we are following the hexagonal-architecture principles.
  • a dependency to the Spring Boot Starter Web , i.e. the starter for building web, including RESTful, applications using Spring MVC, that uses Tomcat as the default embedded container.
  • a dependency to the Spring Boot Starter Actuator , i.e. the tool that helps us to monitor and manage applications. 
    http://localhost:8080/actuator/health
The module includes the following code:
  • HIDemoController.java for Human-Machine interaction, that implements a GET for / and /gui
  • MIDemoController.java for Machine-Machine interaction, that implements a GET for /msg that calls the business logic.
